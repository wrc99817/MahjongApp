# AGENTS.md

## 项目简介

本项目是一个纯血鸿蒙麻将记账 App，暂名“雀账”。V1.0 目标先收口为可独立使用的离线麻将账本：麻将玩家可以在本机创建牌局、维护牌友、手动录入每局四人输赢和共同开销，并查看明细、结算统计和封账历史。好友群组同步账本作为后续在线版本继续迭代。

技术方向：HarmonyOS NEXT Stage 模型、ArkTS、ArkUI。本地优先保存数据，后续接入 HarmonyOS 关系型数据库与 AppGallery Connect 认证/云数据库，实现多端同步与权限控制。

GitHub 仓库：https://github.com/wrc99817/MahjongApp

## 当前计划

- V1.0：离线模式记账闭环，包含创建/编辑牌局、三种记账输入、合计校验、流水、结算、统计、牌友页、我的页、本地持久化、流水撤回/删除、完整历史与流水查看；流水修改功能暂缓。
- V1.5：常用牌友、历史筛选增强、账单分享、离线同步状态提示、本地备份、通知设置和主题切换。
- V2：登录、建群、邀请加入、云备份、多群管理、AppGallery Connect 云同步、权限控制、高级统计、Excel/PDF 导出、会员权益。

## 工作原则

- 保持纯血鸿蒙路线，优先使用 ArkTS、ArkUI、Stage 模型和鸿蒙官方能力；不随意引入跨端框架或非鸿蒙 UI 技术，临时视觉验收除外但必须明确标注。
- UI 遵循鸿蒙原生风格：轻盈、克制、清晰，优先使用 ArkUI 原生控件、圆角层级、模糊材质和自然动效。
- 后续新增页面、组件和视觉特效必须遵循 `prototype/UI_DESIGN.md` 中的“样式特色规则”：轻玻璃导航、柔白卡片、金额语义色、原生短动效、Sheet 编辑形态和可扫读列表，不得随意引入不一致的强主题装饰或炫技动效。
- 所有按钮、可点击小胶囊、可点击列表行和可点击卡片都必须具备轻微按压缩小反馈；后续抽公共组件时应优先在公共 Button、Capsule、ListRow 或 Card 层统一实现，避免入口遗漏。
- 同一页面或同一业务层级中的同等级标题必须采用统一样式，包括字体、字号、字重、行高、容器高度、顶部/底部间距和与卡片边缘的距离；新增或调整卡片时应优先复用统一标题组件，避免各卡片标题视觉起点不一致。
- 同等级页面的元素高度处理方式必须保持一致；同等级、同种类元素也必须采用相同的高度、最小/最大高度、内容自适应或剩余空间分配策略，避免一个元素按内容收敛、另一个同级元素被 `layoutWeight` 等规则撑开造成视觉层级不一致。
- 功能实现优先保证核心记账链路稳定，再推进复杂规则和商业化能力。
- 状态架构遵循鸿蒙 ArkUI 思路：公共业务数据以 `LedgerStore` / `LedgerRepository` 作为唯一事实来源；页面只保留当前 Tab、抽屉、筛选、输入草稿等 UI 状态；牌友、入场金额、当前账面、流水和结算结果应从仓储数据派生，不长期维护重复展示状态。
- 双向绑定只用于表单草稿和父子组件输入联动，例如编辑牌局 Sheet；真实业务数据必须通过统一仓储 action 提交，避免多个组件同时修改同一份事实数据。
- 牌局页、统计页、历史详情等业务展示必须遵循单向数据流：仓储 action 写入 `sessions`、`users`、`rounds` 后，通过统一刷新入口重新读取仓储快照，再由 selector/派生函数计算展示值。不得在页面中长期维护 `setupBuyIn0..3`、`balance0..3` 这类重复事实副本。
- `bindSheet` 半模态中的新建/编辑表单使用临时草稿；点击保存后先一次性提交仓储，再关闭 Sheet，并在关闭动画完成后触发统一刷新。避免半模态背后的页面停留在旧快照，出现切 Tab 后才刷新的问题。
- 如确需刷新触发信号，只允许保存类似 `ledgerViewRevision` 的轻量 UI 版本号；该版本号只用于驱动 ArkUI 重新构建，不得承载业务金额、牌友或流水事实。
- 复杂对象需要响应内部字段变化时，优先使用 ArkUI 的 `@Observed` / `@ObjectLink`；页面树内共享状态可使用 `@Provide` / `@Consume`；`AppStorage` / `LocalStorage` 只用于合适作用域的数据共享，不作为普通事件通知机制滥用。
- 变更代码时必须尽量保持现有无问题功能不受影响；涉及共享模型、状态、导航、账面计算、流水或结算逻辑时，要主动检查相关旧功能是否存在回归风险。
- 开发界面和业务逻辑时，必须严格按照 `prototype/PROTOTYPE_DEV_SPEC.md`、`prototype/prototype-web` 和 `prototype/screenshots` 最新截图执行；如说明书、截图和用户最新批注冲突，必须先向用户确认。
- 调整 Web 原型画布时，页面摆放顺序必须按用户完成核心功能的自然流程组织：先走完牌局创建、记账、编辑和结算，再摆放统计、牌友、我的等后续模块；同一模块的详情页或状态页应紧跟入口页。
- 每次变更 App 代码后，必须重新打包 HAP，并说明最新 HAP 产物路径、生成时间和是否已安装到模拟器；如构建失败，必须明确失败原因。
- 每次变更 App 代码后，必须同步更新 `README.md` 中的功能实现情况，确保项目说明与当前功能状态一致。
- 每次功能变更后，按风险分层验证：低风险文案/样式变更以构建和局部预览为主；中风险 UI 或业务变更验证受影响页面和关键联动；高风险账面、流水、结算、数据模型、同步或导航变更需跑通核心流程和必要回归路径。
- 如本地模拟器验证成本过高，可先交付 HAP 和明确验收清单，请用户在 DevEco Studio / 模拟器中执行验证并提供截图或问题反馈；收到截图后再继续调整。
- 验证过程中产生的临时截图、布局导出或其他中间文件，使用后必须立即删除，不得保存在项目目录中污染仓库。
- 修改数据模型、同步策略、权限逻辑时，要同步更新说明文档或相关注释。
- 任意内容更新后，都要主动判断是否影响相关文档、原型说明、设计基线、README 或项目原则；只有确有影响时才同步更新，避免文档滞后或机械改动。
